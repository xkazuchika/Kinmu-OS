## Context

Kinmu-OS は、従業員100名以下の日本の事業者が自社管理下で利用する、無料かつセルフホスト可能な労務管理ソフトとして新規に構築する。現時点のリポジトリには実装がなく、v0.1では従業員情報、出退勤、残業集計、監査、運用基盤を一貫して提供する。利用者は経営者・労務担当者・従業員であり、個人情報を扱うため、分かりやすさと最小権限を両立する。

制約は、単一事業者向けセルフホスト、少人数運用、GitHub上で公開するオープンソース、従業員100名以下を主な性能規模とすることである。日本語を既定言語とし、組織のタイムゾーンは Asia/Tokyo を既定とするが、明示的に設定可能にする。

## Goals / Non-Goals

**Goals:**

- 1台の小規模サーバーで導入・運用でき、従業員情報と日々の勤怠を一元化する。
- 所有者、労務担当者、従業員に必要な操作だけを安全に提供する。
- 所定勤務時間と実労働時間・残業を追跡可能にし、月次確認とCSV出力を容易にする。
- 信頼感のある、情報密度と可読性のバランスが取れた日本語優先の業務UIを提供する。
- データ所有権、バックアップ、復元、アップグレードを導入者自身が管理できるようにする。

**Non-Goals:**

- 休暇残高・休暇申請・承認、勤怠訂正、メール通知、給与計算、年末調整、社会保険・行政手続きの電子申請、法的な労働時間・有給付与の自動判定。
- 複数社テナントのホスティングSaaS、フレックス・複雑なシフト最適化・深夜勤務、打刻端末や外部SaaSとの双方向同期。
- 初期リリースにおける英語以外の完全な多言語化、ネイティブモバイルアプリ、任意のワークフロー作成機能。

## Decisions

### 1. モジュラーモノリスとして提供する

単一のTypeScript Webアプリケーションを、`identity`、`organization`、`employees`、`attendance`、`reporting`、`audit` のドメインモジュールに分ける。UIと認証済みAPIは同一デプロイメントに置き、PostgreSQLを唯一の永続ストアとする。

- 採用理由: 100名規模では運用対象を最小化でき、トランザクション境界と監査記録を保ちやすい。
- 代替案: マイクロサービスは独立拡張性を得られるが、キュー、監視、認証、デプロイが過剰になるため採用しない。

### 2. Next.js、TypeScript、PostgreSQLを標準スタックにする

Next.js App Router とTypeScriptでサーバーレンダリング中心のWebアプリを実装し、PostgreSQLにマイグレーション管理されたスキーマを保存する。APIはUIと同じ認可サービスを通るJSONエンドポイントとし、外部公開APIは初期リリースでは提供しない。

- 採用理由: フォーム中心の業務UIを高速かつ保守しやすく実装でき、PostgreSQLの整合性制約・トランザクション・JSON監査差分を活用できる。
- 代替案: SPAと別APIサーバーはクライアントとサーバーの分離を強めるが、初期の構成とデプロイを複雑化するため採用しない。

### 3. 単一組織のデータ境界と役割ベース認可を採用する

1インストールを1事業者として扱い、全データを組織設定に論理的に帰属させる。初回セットアップで最初の所有者を作成し、v0.1では `owner`、`hr_admin`、`employee` を定義する。認可は画面非表示だけに依存せず、すべてのサーバー操作で対象従業員・組織・役割を検査する。

- 採用理由: 小規模事業者のv0.1に必要な権限を明確にし、将来の上長承認や複数組織化にもデータモデルを壊さず対応できる。
- 代替案: 管理者・一般利用者の2役割だけでは、所有者権限と日常の労務操作を分離できないため採用しない。

### 4. 個人情報を最小化し、監査証跡を改ざんしにくい追記型で保存する

初期データモデルは、利用目的に不要な個人番号・銀行口座・健康情報を保持しない。資格情報はArgon2idでハッシュ化し、セッションCookieは `HttpOnly`、`Secure`、`SameSite` 属性を使う。監査ログは操作者、時刻、操作、対象、許可された差分の要約を追記型で記録し、通常UIから編集・削除できないようにする。

- 採用理由: 労務データの漏えい範囲を減らし、日常の修正と管理者操作の説明責任を確保する。
- 代替案: 任意フィールドで全情報を保存する方式は柔軟だが、保存すべきでない機微情報が混入しやすいため採用しない。

### 5. 勤怠を原記録と適用日付き勤務ルールで表す

勤怠は従業員・勤務日ごとのレコードとして、出勤・退勤・休憩を保持する。勤務ルールは組織既定または従業員別に、所定開始・終了、所定休憩、1日の所定労働時間、適用開始日を持たせる。打刻は原記録として保持し、日次集計には適用された勤務ルールと実労働時間、所定時間、残業時間を記録する。これによりルール変更を過去の集計へ遡及させず、将来のフレックス・シフト・深夜区分に拡張できる。v0.1の残業は、設定済みの1日所定時間との差分と月次合計を業務確認用に表示し、法令適合の自動判定は行わない。

- 採用理由: 原記録・適用ルール・集計結果を分けることで、月次確認を安全に扱い、後から勤務制度を拡張できる。
- 代替案: 打刻と集計を単一のカレンダーイベントとして持つ方式は、残業と変更履歴を正確に表現しにくいため採用しない。

### 6. 初期データは検証付きCSVで取り込む

労務管理者は、部署と従業員台帳をそれぞれ定義済みのCSVテンプレートで取り込む。各ファイルは取込前に必須列、参照先、形式、重複を検証してプレビューを示し、利用者の確認後にファイル単位で全件を反映する。エラーのあるファイルは一部の行だけを保存せず、行番号と修正内容を返す。取込操作と結果は監査記録に残す。

- 採用理由: 既存の表計算ソフトからの導入を現実的にしつつ、不完全な初期データによる権限・残高・承認経路の破損を防げる。
- 代替案: 任意形式のExcel解析は利用者にとって便利だが、セルフホスト環境での保守・セキュリティ・入力検証が複雑になるため初期リリースでは採用しない。

### 7. 業務UIは共有デザインシステムと段階的開示で構成する

日本語の短いラベル、明確な空状態、確認を要する破壊的操作、テーブルと詳細ペインの一貫したパターンを採用する。従業員向けの打刻・勤務実績確認はモバイルで完結させ、管理画面はデスクトップでの一覧・絞り込み・一括確認を優先する。色だけに状態を依存せず、キーボード操作と適切なラベルをすべての主要フローに設ける。

- 採用理由: SmartHRやジョブカンに期待される業務の明快さを保ちつつ、初見の小規模チームでも学習負荷を抑えられる。
- 代替案: 機能ごとに独自の画面パターンを使う方式は短期的には速いが、製品の一貫性とアクセシビリティを損なうため採用しない。

### 8. コンテナ化された運用と文書化を製品機能として扱う

アプリケーションとPostgreSQLをDocker Composeで起動し、環境変数テンプレート、ヘルスチェック、初回セットアップ、マイグレーション、バックアップ・復元コマンドを提供する。TLSはリバースプロキシで終端する前提を明記し、シークレットをGitに保存しない。ライセンスはAGPL-3.0を採用し、ネットワーク経由で利用する人が実行中バージョンに対応するソースとライセンスへ到達できる導線を設ける。GitHub上にはコントリビューション、脆弱性報告、リリース手順、およびKinmu-OSの名称・ロゴが公式提供元を示すための商標利用ポリシーを整備する。

- 採用理由: セルフホストの自由を保ちながら、改変版をネットワークサービスとして提供する場合にも改変ソースの提供を促し、クローズドな商用フォークを抑制できる。名称・ロゴの利用を分離して定めることで、公式提供元との混同を防げる。
- 代替案: Apache-2.0は広い再利用を促せるが、クローズドな商用フォークも許すため採用しない。商用利用を一律に禁じるソース公開ライセンスは、OSSではなくなり、セルフホスト利用や外部貢献の障壁を高めるため採用しない。

## Risks / Trade-offs

- [法令・就業規則の差異] → 法定計算や適法性の自動保証を対象外と明記し、設定値と出力の最終確認を労務担当者に委ねる。
- [セルフホスト環境のTLS設定不備] → リバースプロキシ例・ヘルスチェック・診断手順を文書化する。
- [権限設定の誤りによる個人情報閲覧] → サーバー側の認可テスト、既定の最小権限、監査ログ、管理画面での可視範囲プレビューを導入する。
- [勤務ルール変更による残業集計の不整合] → 適用開始日と集計時のルール版を保存し、将来の訂正・確定機能を追加しても再集計の対象を追跡できるようにする。
- [不正なCSVによる初期データの破損] → テンプレート、プレビュー、ファイル単位の原子反映、行単位のエラー表示、監査記録を提供する。
- [AGPL-3.0が企業導入や依存関係の選定を難しくする] → ライセンスをREADMEで明示し、依存関係のライセンス確認を継続的インテグレーションに組み込み、将来のデュアルライセンスは著作権管理方針とともに別途検討する。
- [公式版と第三者フォークの混同] → 名称・ロゴの利用ポリシーを公開し、公式配布物・公式ドメイン・公式GitHub組織を明示する。商標登録は公開後の利用状況を踏まえて別途判断する。
- [100名を超える利用や将来的な複数組織] → モジュール境界と組織帰属キーを維持し、負荷・テナント要件が明確になった段階で分離を検討する。
- [初期機能が広すぎること] → v0.1を台帳・打刻・残業集計・CSV・運用の垂直スライスに限定し、休暇・承認・訂正・通知は後続リリースへ分離する。

## Migration Plan

1. Docker Compose、PostgreSQL、環境変数テンプレート、初回所有者セットアップを含む空の初期環境を提供する。
2. バージョン管理したDBマイグレーションをアプリケーションリリースと同時に適用し、適用前にDBバックアップの存在を確認する。
3. リリースごとに互換性、必要な環境変数、手動操作、ロールバック可否をリリースノートに記載する。
4. 失敗時はアプリケーションイメージを直前版へ戻し、破壊的マイグレーションを避ける。不可逆な変更が必要な将来版では、復元手順と事前バックアップを必須化する。
5. 導入前のCSVから部署と従業員を検証環境へ取り込み、結果を確認してから本番取込する手順を提供する。
6. 初期リリース前に、空環境からの導入、CSV取込、バックアップからの復元、メジャー更新の各経路を検証する。
