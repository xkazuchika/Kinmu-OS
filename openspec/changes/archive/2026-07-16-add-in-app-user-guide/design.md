## Context

現在はREADME、勤怠修正の運用文書、画面内の「このソフト」が別々に存在し、利用者が自分の役割で必要な操作を順に学べる入口がない。ガイドはGitHub上でも読めること、セルフホストした実行環境からも同じ内容へ到達できること、外部CMSやネットワーク接続なしに配布できることが必要である。

## Goals / Non-Goals

**Goals:**

- Markdownを詳細内容の基準として、役割別の機能説明・操作手順・トラブル対処・制限事項を保守する
- 認証済みアプリから「このソフト」を経由して、利用者の役割に適したガイドを見つけられるようにする
- セルフホスト用のstandaloneイメージへガイド本文を確実に同梱する
- 320 CSSピクセル、キーボード、見出し構造、現在地、戻り導線を含むアクセシブルな記事表示にする
- リンク切れ、役割別公開範囲、Markdownとカタログの不整合を自動検証する

**Non-Goals:**

- 外部CMS、WYSIWYG編集、利用者による文書編集、コメント機能
- 全文検索エンジン、多言語化、動画チュートリアル、PDF自動生成
- 法律・就業規則への適合性判断や、製品内サポートチケット管理
- 現行機能そのものの挙動変更

## Decisions

### 1. Markdown本文と型付きカタログを分離する

`docs/user-guide/`のMarkdownを本文の基準とし、アプリ側にはslug、タイトル、説明、対象役割、表示順、Markdownファイル名だけを持つ型付きカタログを置く。本文をReactコンポーネントへ重複転記する案は、リポジトリ版とアプリ版が乖離しやすいため採用しない。

### 2. サーバーで限定Markdownを安全に描画する

ガイド画面はサーバーコンポーネントでローカルMarkdownを読み、見出し、段落、箇条書き、番号付き手順、コード、内部リンクに限定した変換を行う。生HTMLは解釈せず文字列として扱う。汎用Markdown依存を追加する案より配布サイズと攻撃面を抑え、利用する記法を文書規約とテストで固定する。

### 3. standalone配布物へガイドを明示的に含める

Next.jsのoutput file tracing設定で`docs/user-guide/**/*.md`をガイドルートへ含め、Dockerビルド後の実行環境でも同じファイルを読めるようにする。ファイルが欠けた場合は空白画面にせず、ガイドを読み込めない旨とGitHub文書への導線を表示する。

### 4. 役割は入口の優先順位に使い、秘密の境界にはしない

従業員には打刻・勤務実績・修正申請を先に、所有者・労務管理者には初期設定・台帳・勤務ルール・審査・CSV・監査を先に表示する。製品操作の説明自体は機密情報ではないため、認証済み利用者は共通機能一覧と他役割の説明も参照できる。権限のない実操作へ誘導する場合は「所有者・労務管理者向け」と明記する。

### 5. 「このソフト」を製品情報とヘルプのハブにする

既存のバージョン・ライセンス・対応ソース表示を維持し、その上部または近接位置に「利用ガイド」への主要導線を追加する。既存ナビゲーション項目を増やし過ぎず、モバイル下部ナビゲーションの3項目も変更しない。

### 6. 文書品質をテスト可能な契約にする

カタログの全slug一意性、全Markdownファイルの存在、必須見出し、内部リンク先、現行バージョンと制限事項の記載を単体テストで検証する。E2Eでは役割別の並び、記事遷移、現在地、320pxの非横スクロール、キーボード到達性、存在しないslugの404を確認する。

## Risks / Trade-offs

- [限定Markdown変換が一般的なMarkdownの全記法を扱えない] → 対応記法を文書化し、未対応記法をテストで拒否する
- [Markdownがstandaloneイメージから欠落する] → tracing設定、Dockerビルド後のガイド表示E2E、欠落時フォールバックを組み合わせる
- [機能変更後に説明が古くなる] → 変更PRのチェック項目、カタログ・リンク・現行バージョン検査、リリース前レビューを追加する
- [役割別表示が権限制御と誤解される] → ガイド上で対象役割を明示し、実際の認可は既存API・画面側に委ねる
- [長い記事がモバイルで読みにくい] → 短い記事へ分割し、目次、前後記事、ガイド一覧へ戻る導線を設ける

## Migration Plan

1. ガイドのカタログ、Markdown読込・描画、テストを追加する
2. `docs/user-guide/`へ機能一覧と役割別記事を追加し、READMEからリンクする
3. `/guide`と`/guide/[slug]`を追加し、「このソフト」から接続する
4. standalone Dockerイメージでファイル同梱と実画面を確認する
5. データ移行は不要。問題があればガイド導線とルートを戻し、Markdown文書はリポジトリ上に保持できる
