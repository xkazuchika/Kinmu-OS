## Context

Kinmu-OSは、打刻を`attendance_events`へ保存し、勤務日ごとの`attendance_days`と`daily_attendance_summaries`を生成している。現状の打刻APIは時系列を検証できるが、確定後の誤りや打刻忘れを直す経路がなく、管理者が変更の根拠と承認者を追跡するためのデータモデルもない。

この変更は従業員画面、管理画面、勤怠集計、権限制御、監査ログ、CSV出力へ横断的に影響する。100名以下の単一組織とセルフホスト運用を前提とし、追加のキュー、通知サービス、外部ワークフロー基盤は導入しない。

## Goals / Non-Goals

**Goals:**

- 従業員が自分の勤務日の有効な打刻列を理由付きで申請できる。
- 審査者が元の打刻列と申請後の打刻列を比較し、安全に承認または却下できる。
- 承認前後の競合を検出し、承認、打刻反映、再集計、監査記録を単一トランザクションで完了する。
- 元の打刻を物理削除せず、どの申請で置き換えられたか追跡できる。
- 既存の役割モデル、組織分離、日本語優先UI、320px対応を維持する。

**Non-Goals:**

- 有給休暇、欠勤、休日出勤、残業事前申請など勤怠修正以外の申請種別。
- 多段階承認、承認経路設定、代理承認、部署長ロール。
- メール、Slack、プッシュ通知。
- 月次締め、締め解除、給与計算、法令適合の自動判定。
- 管理者が申請を介さず本人の打刻を直接編集する機能。

## Decisions

### 1. 申請は操作差分ではなく「希望する有効打刻列」を保持する

`attendance_correction_requests`に組織、従業員、勤務日、理由、状態、申請者、審査者、審査コメント、審査日時、基準リビジョンを保存する。子テーブル`attendance_correction_entries`には、申請時点の元打刻スナップショットと、申請後に有効としたい打刻列を種別付き・順序付きで保存し、対応する元打刻がある場合はそのIDも保持する。

完全な希望打刻列を保存すると、追加・変更・削除を同じモデルで表現でき、審査画面とサーバーが同じ時系列検証を共有できる。操作ごとのJSONパッチは小さいが、適用順序と途中状態が複雑になるため採用しない。

### 2. 勤務日リビジョンで楽観的競合検出を行う

`attendance_days`へ単調増加する`revision`を追加し、通常打刻と承認済み修正の反映ごとに増やす。既存勤務日がない申請は基準リビジョン0として扱う。申請は作成時のリビジョンを保持し、承認時に従業員IDと勤務日をキーにPostgreSQL advisory transaction lockを取得して、現在値との一致を検証する。

一致しない場合はHTTP 409相当の競合として承認を中止し、申請を`pending`のまま残す。利用者は最新記録を確認して申請を取り消し、再申請する。自動マージは意図しない時刻を確定する危険があるため行わない。

### 3. 元打刻は保持し、有効な打刻セットだけを切り替える

`attendance_events`へ`correction_request_id`と`superseded_by_correction_request_id`を追加する。承認時は現在有効な打刻を承認申請で置換済みとしてマークし、申請の希望打刻列から新しいイベントを`source = correction`で挿入する。有効な打刻取得は`superseded_by_correction_request_id IS NULL`を条件とする。

物理更新だけで時刻を書き換える方式より元記録の復元性と説明可能性が高く、イベントを全てコピーするコストは対象規模では十分小さい。却下・取消では打刻を変更しない。

### 4. 状態遷移を限定し、審査結果を不変にする

申請状態は`pending`から`approved`、`rejected`、`cancelled`のいずれか一度だけ遷移する。従業員は自分の`pending`申請だけを取り消せる。`owner`または`hr_admin`は同一組織の`pending`申請だけを審査でき、申請者本人は審査できない。却下コメントは必須、承認コメントは任意とする。

同じ従業員・勤務日に複数の`pending`申請を許さず、二重審査は状態条件付き更新で防ぐ。承認済みの内容をさらに直す場合は、現在の有効打刻を基準とする新しい申請を作成する。

### 5. 検証、反映、集計、監査を一つのドメインサービスに集約する

申請作成時と承認時の両方で、組織タイムゾーン、勤務日、打刻種別、時刻順、`clock_in`から始まる状態遷移、休憩の対、最大1回の`clock_out`を検証する。承認トランザクションは勤務ルールの解決、勤務日の作成または更新、有効打刻の切替、日次状態・集計の再生成、リビジョン増加、申請状態更新、監査ログ追加をまとめて行う。

集計処理は通常退勤と修正承認で共有できる純粋な再計算関数へ抽出する。希望打刻列が退勤で終わらない場合は勤務日を`open`とし、日次集計を未確定状態へ戻す。

### 6. APIとUIは役割別に分離する

従業員向けAPIは自分の申請作成、一覧、詳細、取消だけを提供する。管理APIは組織内申請の一覧、詳細、承認、却下を提供し、すべての読み書きで組織IDと役割をサーバー側検証する。

従業員の勤務実績には日別の「修正を申請」導線と申請状態を表示する。管理者ナビゲーションには申請一覧を追加し、ホームに未処理件数を表示する。審査画面は元／申請後の打刻を時刻順に並べ、追加・変更・削除を色だけに依存せず文言と記号で示す。

## Risks / Trade-offs

- [古いコードが置換済み打刻を集計する] → 有効打刻取得を単一ヘルパーへ集約し、一覧、状態取得、集計、CSVの統合テストを追加する。
- [承認と通常打刻が競合する] → 同じ従業員・勤務日のadvisory lockとリビジョン一致を必須にし、競合時は一切変更しない。
- [申請した時刻列が不正で勤務状態を壊す] → 作成時と承認時に共通状態機械で再検証し、DB値をクライアント検証だけに依存しない。
- [却下・承認済み申請が書き換えられる] → 状態条件付き更新、追記型監査、審査後フィールドの通常API非公開で不変性を守る。
- [イベントコピーによりデータ量が増える] → 1勤務日のイベント数は小さく、対象規模では追跡性を優先する。組織・勤務日・有効状態の索引を追加する。
- [管理者本人による自己承認が必要な小規模運用] → 改ざん防止を優先して禁止する。単独運用者の自己勤怠修正は今回の対象外として明示する。

## Migration Plan

1. 申請状態enum、申請・希望打刻テーブル、勤務日リビジョン、打刻の修正参照列と索引を追加する。既存勤務日のリビジョンは0で初期化し、既存打刻は全て有効なままとする。
2. 有効打刻取得と日次再計算を共通化し、既存打刻・一覧・CSVが移行前と同じ結果になる回帰テストを通す。
3. 申請APIと審査API、監査操作種別を追加する。
4. 従業員画面、管理者画面、未処理件数を段階的に公開する。
5. ロールバック時は新UIとAPIを停止する。追加テーブルと列はデータ保持のため残し、旧コードは`superseded_by_correction_request_id IS NULL`の打刻だけを読む互換状態を維持する。
