## Context

Kinmu-OS v0.4は、Drizzle ORMとPostgreSQLを使い、勤務カレンダー、打刻、勤怠修正、休暇、月次締め、CSV、監査を一つのセルフホスト構成で提供している。勤怠の実績残業は計算できるが、残業・休日出勤を事前に申し出て審査し、承認予定と実績を照合するデータと画面はない。

この変更は、従業員、所有者、労務管理者を主な利用者とする100名以下の組織を対象に、既存の組織境界、月単位のPostgreSQL advisory lock、追記型監査、月次締めリビジョンへ申請フローを統合する。承認は勤怠実績や法令判定ではなく、社内運用上の予定確認である。外部通知サービスを必要とせず、オフラインのセルフホスト環境内で完結させる。

## Goals / Non-Goals

**Goals:**

- 適用日付きポリシーに基づく残業・休日出勤の申請、単段階審査、取消を安全に提供する
- 承認予定と現在の打刻実績を同じ規則で照合し、日次、月次、ダッシュボード、CSV、締めへ一貫して反映する
- 申請・審査と通知・監査を原子的に保存し、組織境界と本人範囲をAPIで強制する
- 締め済み月を不変にしつつ、v0.4組織の既存結果を自動変更しない移行を行う
- 320 CSSピクセル以上、キーボード、支援技術で利用できる申請・審査・通知UIを既存App Shellへ統合する

**Non-Goals:**

- 36協定、法定休日、割増賃金、健康確保措置などの法令適合の自動判定
- 給与計算、残業代計算、代休・振替休日の自動付与
- 多段承認、承認経路の組織別設計、代理申請、代理承認
- メール、Slack、Webhook、プッシュ通知などの外部配信
- 承認予定分数による打刻または勤怠実績の自動生成

## Decisions

### 1. 適用日付きポリシーを独立した履歴として保存する

`overtime_request_policies`に組織、適用開始日、状態、申請時刻の入力単位、事前申請要否、実績差異の許容分数、未解決差異を締め阻害にする設定、バージョン、作成・有効化情報を保存する。入力単位は`1`、`5`、`10`、`15`、`30`分から選択し、開始・終了時刻が単位境界に一致することを検証する。事前申請要否が有効な場合は予定開始日時以後の新規提出を拒否し、無効な場合は未締め期間への事後申請を許可する。

更新時は既存行を上書きせず、新しい適用開始日の行を作る。対象日に有効な、状態が`active`で適用開始日が最も新しい行を選ぶ。移行時は組織ごとに未有効の`draft`を一件作り、管理者が影響範囲を確認して有効化するまで申請必須化、差異判定、締め阻害を開始しない。

代案として組織テーブルへ現在値を直接追加する方法は単純だが、過去日の再現と締めスナップショットの根拠が失われるため採用しない。既存組織で自動有効化する方法も、アップグレード直後に過去月の締め可否を変えるため採用しない。

### 2. 申請は一勤務日・一時間帯の不変な予定を中心にする

`overtime_work_requests`に組織、従業員、申請者、区分、勤務日、組織タイムゾーンで解釈した開始・終了日時、予定休憩分数、予定分数、理由、ポリシー参照、申請時の勤務ルール・カレンダー根拠の要約、状態、楽観ロック用バージョン、審査者・日時・コメントを保存する。区分は所定勤務日なら`overtime`、所定休日または会社休日なら`holiday_work`とし、従業員入力ではなくカレンダー判定から検証する。

時刻入力は組織ローカル日時として受け取り、UTCの`timestamp with time zone`へ変換して保存する。終了は開始より後かつ24時間以内で、翌日にまたがっても元の`work_date`に帰属する。予定分数は経過分数から休憩を差し引き、正値でなければならない。所定勤務日の申請時間帯には所定時間外が含まれる必要がある。審査待ちまたは承認済みの時間帯同士は重複を許さず、DB制約で表現できる範囲は制約を使い、同時作成は従業員・勤務日のadvisory lockとトランザクション内再検査で防ぐ。

状態は`pending`から`approved`、`rejected`、`cancelled`の一つへだけ遷移する。内容の直接編集は提供せず、審査待ちを取り消して再提出する。これにより審査・通知・監査が参照した内容を不変に保つ。

代案の複数日・複数時間帯を一申請にまとめる形式は入力件数を減らせるが、休日区分、締め月、重複、実績差異の扱いが複雑になるため初期版では採用しない。

### 3. 審査と月次変更は既存の月ロック順序へ統合する

申請提出、取消、審査、勤怠修正、月次締めは、最初に組織・対象月のadvisory transaction lockを取得し、その後に申請行または従業員・勤務日のロックを取得する。複数月を扱う場合は`YYYY-MM`昇順で取得する。審査APIはクライアントの`expectedVersion`を必須とし、月の締め状態、申請状態、在籍、自己承認、対象日のポリシー、時間帯重複をロック取得後に再検査する。

申請状態、審査情報、通知、監査は一つのDBトランザクションで保存する。HTTPでは未認証を401、権限不足または範囲外を情報露出しない403/404、古いバージョンや同時変更を409、業務入力エラーを422として既存API規約に合わせる。

代案のアプリケーション内mutexは複数コンテナ構成で共有できない。申請行だけのロックは月次締めとの順序を保証できないため採用しない。

### 4. 実績差異は未締め月では動的計算し、締め時だけ固定する

未締め月の差異は、現在有効な打刻・勤怠修正、承認済み申請、対象日に有効なポリシーから共通ドメイン関数で計算する。同一勤務日に複数の重複しない承認申請がある場合は予定分数を合計する。所定勤務日は既存の日次残業分数、所定休日・会社休日は休憩控除後の実労働分数を実績として使う。

判定規則は次の順で固定する。

- 実績があり承認申請がない場合は`unapproved_actual`
- 承認申請があり実績が0の場合は`no_actual`
- 実績が予定を許容分数より超える場合は`exceeded_request`
- 実績が予定より許容分数を超えて少ない場合は`under_request`
- それ以外は`within_request`

月次締めでは、申請ID・区分・状態・予定分数、実績分数、差分、差異状態、ポリシー参照と表示用要約を締めリビジョンのJSONスナップショットへ保存する。締め済み一覧とCSVは動的再計算せず、この値を使う。通知は業務イベントに限定し、打刻のたびに差異通知を大量生成しない。差異は勤務実績、管理一覧、ダッシュボードで確認する。

代案の差異テーブルを打刻ごとに更新する方式は検索を単純にするが、修正・休暇・カレンダー変更との同期漏れが起きやすい。100名・31日程度は索引付き動的計算で扱えるため、初期版では締めスナップショット以外を派生値とする。

### 5. 締め阻害はポリシーと対象日ごとに判定する

`pending`の残業・休日出勤申請は、ポリシー設定に関係なく対象月の締めを阻害する。有効ポリシーが未解決差異の阻害を有効にしている対象日は、`exceeded_request`、`no_actual`、`unapproved_actual`を阻害する。`under_request`は実績が確定しているため一覧には出すが、初期版では締めを阻害しない。ポリシーがない日と未有効ドラフトだけの組織は、残業差異による締め阻害を追加しない。

締め済み月では申請の提出・取消・審査と、その月へ効力を持つポリシー変更を拒否する。変更が必要な場合は既存の理由付き再開と再締めを使用する。

代案の全差異を常に阻害する方式は厳格だが、既存組織の移行と不足実績の運用に過剰な負担を生むため採用しない。

### 6. 通知は対象参照型のアプリ内受信箱にする

`notifications`に組織、受信利用者、種別、件名、最小限の要約、対象種別、対象ID、作成日時、既読日時を保存する。任意URLや他者が指定した受信者を保存せず、サーバーが申請イベントと現在の同一組織の有効な役割から受信者を決める。申請提出は有効な所有者・労務管理者、審査結果は申請者、取消は審査権限者を受信者とする。

対象画面への経路は対象種別とIDからサーバー側の固定マッピングで生成し、遷移前に現在の権限を再検査する。過去通知の受信者が役割を失った場合は最小限の通知履歴だけを見せ、管理者用詳細を開かない。未読件数と一覧向けに`(organization_id, recipient_user_id, read_at, created_at)`索引を作る。既読化は本人の通知だけを個別または表示済みID集合で更新する。

代案のURL保存はopen redirectや古い権限を前提にした導線を生む。メール通知はセルフホスト利用者にSMTP設定と配信失敗管理を要求するため対象外とする。

### 7. APIと画面は既存App Shellへ役割別に追加する

ドメイン処理は`src/lib`のポリシー、申請、差異、通知モジュールに分け、Route Handlerは認証、入力検証、ドメイン呼び出し、既存形式のエラー応答に限定する。組織ID、本人の従業員ID、通知受信者は認証済みactorから導出し、クライアント値を信頼しない。

従業員向けに`/overtime`で新規申請、プレビュー、履歴、取消、実績差異を提供する。管理者向けに`/overtime/reviews`と`/overtime/settings`を提供する。全役割に`/notifications`とApp Shellの未読バッジを提供する。既存の勤務実績、月次勤怠、ダッシュボード、締め、CSV、監査へ申請状態と差異フィルターを追加する。カードと詳細行を使い、320pxで表全体の横スクロールを要求しない。状態は文言とアイコンで示し、確認ダイアログはフォーカストラップ、戻りフォーカス、結果のlive regionを持つ。

### 8. バージョンと文書をv0.5.0として一緒に更新する

アプリ内ガイドのMarkdown、記事カタログ、READMEの機能表、導入・更新手順、APIや運用上の制限、変更履歴、`package.json`の製品バージョンをv0.5.0へ揃える。ガイドには申請承認が法令適合を保証せず、承認分数が実績へ加算されないことを明記する。既存のガイド整合性テストに新記事、内部リンク、対象バージョンを追加する。

## Risks / Trade-offs

- [カレンダーや勤務ルールの変更で申請時の区分と現在の区分が変わる] → 申請時の根拠を保存し、審査時は現在値を再検査して差異を表示する。締め時は現在有効な勤怠判定と参照を固定する
- [翌日またぎと夏時間で予定分数が誤る] → 組織ローカル日時の存在・曖昧性を検証してUTCへ変換し、24時間上限と勤務日帰属をテストする
- [締め、審査、修正の同時実行がデッドロックする] → 月ロックを先、月を昇順、対象行を後という取得順序を共通ヘルパーと統合テストで固定する
- [全管理者への通知で件数が増える] → 通知はイベント単位の短い行に限定し、受信者・未読索引とページネーションを使う。初期対象の100名規模で負荷試験する
- [動的差異計算が一覧を遅くする] → 対象月・組織・従業員・状態の索引、月単位の一括取得、共通集計でN+1を避ける。代表的な100名×31日データで応答時間を計測する
- [承認を法的な残業許可と誤認する] → 申請、差異、CSV、ガイドに業務確認用であることを表示し、法定区分や割増率を返す項目を設けない
- [通知要約から退職者や権限外の情報が漏れる] → 要約を最小化し、受信者・組織スコープを全取得に強制し、詳細遷移時に現在権限を再検査する

## Migration Plan

1. v0.4データを使う移行テストとバックアップ復元テストを先に用意する。
2. 加算的なDrizzle migrationでポリシー、申請、通知、必要な締めスナップショット項目、制約・索引を追加する。
3. 各既存組織へ未有効の初期ドラフトを作成する。既存の打刻、集計、締めリビジョンは更新しない。
4. ドメイン、API、勤怠・締め・CSV統合、UI、ガイドを配布し、ポリシー未有効時にv0.4回帰テストが通ることを確認する。
5. 管理者が設定画面で影響開始日と締め阻害設定を確認して明示的に有効化する。
6. 本番相当Docker Composeでv0.4バックアップから更新し、ログイン、既存締め結果、申請、審査、通知、再起動後の永続化、CSVをスモークテストする。

ロールバックは、アプリをv0.4へ戻す前にv0.5で作成された申請・通知を読み取れないことを周知し、原則として更新前バックアップを別環境へ復元する。DB列・テーブルの破壊的なdown migrationは提供せず、v0.5テーブルを残したままアプリだけを戻す場合も、v0.4で対象月を変更してから再度v0.5へ進めない運用とする。

## Open Questions

初期実装を開始するための未決事項はない。多段承認、代理申請、差異イベント通知、メール等の外部通知、代休・振替休日、法令判定は将来の独立した変更として検討する。
