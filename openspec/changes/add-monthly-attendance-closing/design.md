## Context

現在のKinmu-OSは、有効打刻と適用日付き勤務ルールから日次・月次集計を再計算し、承認済み勤怠修正を同じ集計とCSVへ反映する。対象月を確定するデータ境界はなく、勤務ルール、従業員台帳、承認済み打刻が後から変わると、以前に出力した月次結果を同じ条件で再現できない。

月次締めは、打刻、修正承認、集計、CSV、監査、利用ガイドを横断する。100名・31日程度を一つのPostgreSQLトランザクションで扱い、セルフホスト環境だけで確定結果を保持する必要がある。

## Goals / Non-Goals

**Goals:**

- 組織タイムゾーンで終了した月だけを、安全な事前検査後に締める
- 締め時点の従業員表示情報、日次集計、月次合計、勤務ルール参照をリビジョン付きで再現可能にする
- 締め処理と打刻・修正承認の競合を単一のロック規約で防ぐ
- 締め済み月への変更を拒否し、理由付き再開後だけ修正を許可する
- 締め、再開、再締めと確定CSV出力を監査可能にする
- 既存の月次一覧、勤務実績、CSV、利用ガイドへ自然に統合する

**Non-Goals:**

- 給与計算、振込、年末調整、法令適合判定
- 休日・祝日・有給休暇・欠勤予定から「打刻がない日」を自動判定すること
- 部署別締め、従業員別締め、複数段階の締め承認
- 自動締め、締め期限通知、外部給与ソフトAPI連携
- 締め済みスナップショットの削除または上書き

## Decisions

### 1. 終了済み暦月だけを組織単位で締める

対象月の翌月1日が組織タイムゾーンで到来した後だけ締めを許可する。現在月を途中で締める案は、通常打刻との競合と運用ミスを増やすため採用しない。初期版は組織全体を単位とし、部署・個人単位の部分締めは扱わない。

### 2. 可変な期間状態と不変な締めリビジョンを分離する

組織と対象月に一意な期間行へ`open`または`closed`、次のリビジョン番号、更新用バージョンを保持する。締めるたびに不変の締めリビジョンを追加し、再開時はそのリビジョンへ再開者、日時、理由を追記して期間を`open`へ戻す。以前のリビジョンは削除せず、再締めで新しいリビジョンを作る。

単一行の締めフラグだけを保存する案は、過去の締め結果と再締め差分を再現できないため採用しない。

### 3. 締め時点のCSVに必要な値をスナップショットする

各締めリビジョンに、対象従業員の社員番号・表示名・部署コード・部署名、勤務日、状態、有効打刻の参照、実労働・所定・残業時間、適用勤務ルールの識別子と表示名を保存し、従業員別・組織全体の月次合計も固定する。締め済み一覧とCSVは現在の台帳やルールを再結合せず、このスナップショットを読む。

集計値だけ保存して表示名を現在の台帳から読む案は、過去CSVが台帳変更で変わるため採用しない。元打刻と勤務ルールの完全コピーは重複が大きいため、既存履歴への識別子参照と出力に必要な値の固定を組み合わせる。

### 4. 月単位のトランザクションロックを全変更経路で共有する

締め、再開、通常打刻、勤怠修正申請、取消、承認反映は、組織IDと対象月から導くPostgreSQLトランザクションアドバイザリロックを最初に取得する。締め処理はロック取得後に未退勤と`pending`申請を再検査し、スナップショット、期間状態、監査ログを単一トランザクションで保存する。既存の勤務日ロックは月ロックの後に取得し、ロック順を固定する。

アプリケーション内Mutexは複数コンテナで機能せず、期間行だけのロックは初回締め前に行がない競合を扱いにくいため採用しない。

### 5. 締め前の阻害要因は検出可能な不整合に限定する

対象月の未退勤勤務日、審査待ち勤怠修正、集計不能な勤務日を阻害要因として件数と対象を返す。休日・休暇カレンダーがない現行版では、打刻が一件もない日を欠勤や漏れと判定しない。画面は締め前検査結果から該当する勤怠・申請一覧へ移動できるようにする。

### 6. 再開は理由必須で、再締めは新しいリビジョンにする

所有者・労務管理者は具体的な理由を入力し、確認ダイアログを経て再開できる。再開後は通常の変更経路を再び許可し、再締め時に事前検査と全スナップショットを作り直す。古い確定CSVを黙って置き換えず、リビジョン番号で区別する。

### 7. v0.3へ一貫して更新する

パッケージ、環境変数既定値、Compose、README、アプリ内表示、全ガイドの対象バージョンを`0.3.0`へ更新する。利用ガイドでは月次締めを現行機能へ移し、多段承認と代理申請は未対応のまま残す。

## Risks / Trade-offs

- [締め中に通常打刻や承認が競合する] → 全変更経路で月ロックを先に取得し、締め状態を同じトランザクション内で再確認する
- [スナップショットが増える] → 対象を100名以下・日次出力値に限定し、月・リビジョンの索引と性能テストを追加する
- [打刻なしを検出できず不完全な月を締める] → 現行版で検出可能な範囲を明記し、休日・休暇管理導入後に事前検査を拡張する
- [再開後のCSVと以前のCSVが混同される] → CSVへ締め状態、締め日時、リビジョンを含め、監査ログから出力条件を追跡する
- [機能ロールバック後に締め済みデータを変更できる] → マイグレーションは加算のみとし、旧アプリへ戻す前に運用を停止する。DB表は削除しない

## Migration Plan

1. 期間、締めリビジョン、日次・月次スナップショットの加算マイグレーションを適用する
2. 既存の過去月は自動締めせず`open`として扱い、管理者が確認後に初回締めする
3. 月ロックと締め状態検査を既存変更APIへ追加してから、締めAPIとUIを公開する
4. CSV、監査、ガイド、バージョンを同じリリースで更新する
5. 問題時は締めUIを無効化してアプリを戻すが、作成済み締め表と履歴は保持する。締め済み期間がある環境を締め検査のない旧版で運用しない

## Open Questions

- 休日・休暇管理を追加した時点で、打刻なし予定勤務日を締め阻害要因へ含める
- 将来の給与連携では、締めリビジョンを外部連携単位とし、連携済み後の再開に追加確認を求めるかを別変更で決める
