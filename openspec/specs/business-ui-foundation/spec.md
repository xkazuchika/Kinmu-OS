# business-ui-foundation

## Purpose

従業員と管理者が迷わず業務を完了できるよう、日本語優先で役割に適した情報設計、レスポンシブ表示、アクセシビリティ、操作フィードバックの共通要件を定義する。

## Requirements

### Requirement: 一貫した日本語優先の業務UI
システムは、主要画面で共有のナビゲーション、見出し、フォーム、一覧、状態表示、確認ダイアログを使用し、日本語を既定の表示言語としなければならない（MUST）。各画面は現在地、主な次の操作、空状態またはエラー時に取るべき行動を明確に表示しなければならない（MUST）。

#### Scenario: データがない一覧を開く
- **WHEN** 利用者が対象データのない従業員一覧または申請一覧を開く
- **THEN** システムは空状態の理由と、権限のある利用者が次に実行できる操作を表示する

### Requirement: 役割に適したホーム画面
システムは、従業員に打刻、当月の勤務実績、残業時間を優先表示し、所有者・労務担当者には未退勤と管理対象の概要を優先表示しなければならない（MUST）。利用者は権限のない管理機能をナビゲーションから実行できてはならない。

#### Scenario: 従業員がホーム画面を開く
- **WHEN** 従業員が認証後にホーム画面を開く
- **THEN** システムは自分の打刻操作、直近の勤務実績、当月の残業時間を表示する

#### Scenario: 労務担当者がホーム画面を開く
- **WHEN** 労務担当者が認証後にホーム画面を開く
- **THEN** システムは未退勤と従業員別の残業集計を表示する

### Requirement: レスポンシブかつアクセシブルな操作
システムは、従業員の打刻・勤務実績確認・プロフィール確認を幅320 CSSピクセル以上の画面で横方向スクロールなしに完了できなければならない（MUST）。主要操作はキーボードだけで操作でき、入力には可視ラベルとエラーメッセージを関連付け、状態を色だけで伝えてはならない。

#### Scenario: モバイル幅で打刻する
- **WHEN** 従業員が幅320 CSSピクセルの画面で打刻画面を開く
- **THEN** システムは横方向スクロールを必要とせずに現在の状態確認と打刻を可能にする

#### Scenario: キーボードでフォームを送信する
- **WHEN** 利用者がキーボードだけで主要な申請フォームを操作する
- **THEN** システムはすべての入力、エラー確認、送信操作へ到達できるようにする

### Requirement: 破壊的操作と送信結果のフィードバック
システムは、利用者無効化、退職処理、勤務ルール変更などの影響が大きい操作について、対象と影響を示す確認を求めなければならない。システムは保存・送信の成功、失敗、未送信状態を利用者が識別できる形で表示しなければならない（MUST）。

#### Scenario: 利用者を無効化する前に確認する
- **WHEN** 所有者が利用者の無効化を選択する
- **THEN** システムは対象利用者とログイン停止の影響を示す確認操作を表示する

#### Scenario: プロフィール更新に失敗する
- **WHEN** 従業員がプロフィール更新を送信し、サーバー側の検証で失敗する
- **THEN** システムは入力内容を保持し、修正が必要な項目と失敗理由を表示する
