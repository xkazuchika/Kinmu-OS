## リリースノート

v0.5.0では、適用日付きポリシーに基づく残業・休日出勤申請を追加した。従業員は勤務予定、予定時刻、休憩、理由を確認して申請でき、所有者または労務管理者は自己承認を除く単段階審査を行える。申請の提出、取消、承認、却下は監査ログとアプリ内通知へ同じトランザクションで保存される。

承認済みの予定分数と現在の打刻実績は、申請内、申請未満、申請超過、実績なし、未申請の実績として比較される。差異は勤怠一覧、ダッシュボード、月次締め、UTF-8 CSVへ統合し、締め済み月ではリビジョンの固定スナップショットを使用する。承認分数を実労働へ加算せず、36協定、法定休日労働、割増賃金等の法令適合を自動判定しない。

通知はセルフホスト環境内で完結し、任意URLを保存しない。対象種別とIDから固定経路を生成し、通知を開く時点の役割と組織を再検査する。申請、審査、設定、通知はデスクトップと320 CSSピクセル幅で操作でき、確認ダイアログのフォーカストラップ、Escape、起点へのフォーカス復帰、キーボード操作を含む。

## v0.4バックアップからの移行・復元

1. `v0.4.0`タグの本番Composeを独立volumeとポートで起動し、所有者を有効化した。
2. 2026-06の締め済みrevision 1と、所定480分、実働540分、残業60分の日次スナップショットを保存した。
3. `scripts/backup.sh`でPostgreSQLカスタム形式バックアップを取得した。
4. 別の空volumeへ`CONFIRM_RESTORE=EMPTY_DATABASE scripts/restore.sh`で復元した。空でないDBへの復元拒否もスクリプトの事前検査で維持している。
5. v0.5のmigratorを適用し、migration 16件、既存の締め状態・revision・集計値・表示根拠が不変であることを確認した。
6. 既存組織へ未有効の残業申請ポリシードラフトが1件だけ作成され、既存所有者のログインがHTTP 200で成功した。
7. v0.5本番Composeで全9件のE2Eを実行し、アプリ再起動前後で残業申請、通知、締め済み月、承認監査の件数が一致した。
8. 再起動後に既存所有者でログインし、復元した2026-06の確定CSVから従業員番号、所定480分、実働540分、残業60分、revision 1を確認した。

## 更新とロールバック

更新前に、締め・再開・再締めを完了して確定CSVを保管し、次を実行する。

```sh
BACKUP_DIR=/secure/backups scripts/backup.sh
scripts/update.sh
```

更新後はヘルスチェック、既存所有者のログイン、直近の締め状態・revision・確定CSVを先に確認する。その後、残業申請設定の未有効ドラフトを開き、適用開始日、入力単位、事前申請、許容分数、締め阻害設定を確認して明示的に有効化する。有効化前はv0.4の勤怠・締め挙動を維持する。

確実な切り戻しは、更新前バックアップを新しい空DBへ復元し、v0.4アプリを接続する。破壊的なdown migrationは提供しない。実地確認では、v0.5データを持つDBへv0.4アプリを接続してヘルスチェックと既存ログインは成功したが、残業申請APIは404となり、v0.5の申請・通知を閲覧・運用できなかった。

そのため、次を守る。

- 切り戻し判断中は新規入力を停止する。
- v0.5で作成した申請、通知、監査、ポリシーをv0.4で変更しない。
- v0.4アプリへ戻した状態で対象月を変更してから、同じDBを再度v0.5へ進めない。
- v0.5で作成したデータを保持したまま旧版を継続運用せず、原則として更新前バックアップを空DBへ復元する。
- 更新後の入力は更新前バックアップへ自動的に取り込めないため、必要な再入力と監査上の扱いを決めてから切り戻す。

## 既知の制限

- 36協定、法定休日、割増賃金、健康確保措置などの法令適合は自動判定しない。
- 承認済み予定分数は実労働・残業へ加算しない。
- 多段承認、代理申請・代理承認、メール、Slack、Webhook、プッシュ通知は未対応。
- 給与計算、残業代計算、代休・振替休日の自動付与は未対応。
- v0.5で作成した申請・通知はv0.4アプリから利用できない。
