## マイグレーション・復元

- 空のPostgreSQL 16へ本番Composeを起動し、15件のマイグレーション、v0.4テーブル、アプリのヘルスチェックを確認した。
- `v0.3.0`タグの本番migratorで作成したDBへ、既存組織と締め済み日次スナップショットを保存してからv0.4本番migratorを適用した。
- v0.3組織には未有効の平日勤務ドラフトが1件追加され、締め済みの所定480分・実働480分は変化しなかった。
- `scripts/backup.sh`で取得したカスタム形式バックアップを、別ボリュームの空DBへ`scripts/restore.sh`で復元し、15件の履歴とv0.4テーブルを確認した。

## 更新後の勤務カレンダー有効化

1. 更新前に`scripts/backup.sh`でバックアップを取得する。
2. `scripts/update.sh`でイメージをビルドし、マイグレーションとアプリ更新を行う。
3. 労務管理者が「勤務カレンダー」を開く。v0.3から更新した組織では、月〜金を勤務日とするドラフトが未有効で表示される。
4. 適用開始日、曜日、会社休日、従業員例外と影響人数を確認する。
5. 締め済み期間より後の日を適用開始日にして有効化する。有効化前の勤怠や締めリビジョンへは遡及しない。
6. 勤怠一覧の未解決・競合、休暇残高、月次締め前検査を確認してから通常運用へ移る。

## 既知の制限

- 時間単位休暇には未対応。休暇単位は全日または半日。
- 多段承認、代理申請、メール・アプリ内通知には未対応。
- 法定休日、法定外休日、割増賃金、休暇付与義務などの法令適合性は自動判定しない。
- 残業・休暇・欠勤の一覧とCSVは運用支援情報であり、法令適合の判定結果ではない。
- 勤務カレンダーの有効化前は、v0.3互換の勤怠表示と締め阻害範囲を維持する。

## ロールバック時の注意

- v0.4のDB変更は既存データを保持する加算マイグレーションで、down migrationは提供しない。
- アプリだけを旧版へ戻す前に、旧版がv0.4スキーマ上で動作することを検証する。確実な復旧が必要な場合は、更新前バックアップを新しい空DBへ復元して旧版を接続する。
- `scripts/restore.sh`は空DB以外への復元を拒否する。稼働DBへ上書き復元しない。
- 更新後に作成された休暇申請・台帳取引を、更新前バックアップへ戻して保持することはできない。切り戻し判断中は新規入力を停止する。
- カレンダー設定を誤って有効化した場合も過去レコードを直接書き換えず、新しい適用日付きパターンまたは例外で訂正する。締め済み月は必要に応じて理由付きで再開する。
